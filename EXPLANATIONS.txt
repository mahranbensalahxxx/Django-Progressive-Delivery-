SECTION 1: WHY DO WE HAVE 13 PODS? (Pod Count Explanation)
================================================================================

When you run:
  kubectl get pods -n progressive-django

You see 13 pods. Here's WHY each group exists:

3x django-blue-xxxxx     → Blue-Green strategy (Student 2)
   These are the STABLE v1 pods. We set "replicas: 3" in blue-deployment.yaml
   because in production you want multiple copies for high availability.
   If one pod crashes, the other 2 keep serving. 3 is a common minimum
   for production workloads.

3x django-green-xxxxx    → Blue-Green strategy (Student 2)
   These are the NEW v2 pods. Also "replicas: 3" in green-deployment.yaml.
   Both Blue and Green run AT THE SAME TIME. The Service decides which
   group gets real traffic. The other group is idle but ready.
   This is what makes Blue-Green "zero downtime" — the switch is instant
   because both versions are already running.

5x django-rollout-xxxxx  → Canary strategy (Student 1 — Argo Rollouts)
   These are managed by Argo Rollouts, NOT a regular Kubernetes Deployment.
   We set "replicas: 5" in rollout-canary.yaml because canary needs more
   pods to split traffic in percentages:
     - 20% traffic to v2 = 1 out of 5 pods runs v2 (1/5 = 20%)
     - 50% traffic to v2 = 2-3 out of 5 pods run v2
     - 100% = all 5 pods run v2
   With only 3 pods, you can't do 20% — the math doesn't work cleanly.
   5 pods gives Argo clean percentage steps.

2x nginx-shadow-xxxxx    → Shadow Traffic (Student 3)
   These run NGINX as a reverse proxy. "replicas: 2" in nginx-deployment.yaml.
   2 is enough for a demo proxy — it just forwards requests, it doesn't
   run heavy app logic. One would work, but 2 gives basic redundancy.

SECTION 2: WHY DO WE HAVE 5 SERVICES?
================================================================================

When you run:
  kubectl get services -n progressive-django

You see 5 services:

django-service (port 80 → 8000)
   The MAIN service for Blue-Green. Its "selector" field determines which
   pods get traffic. selector: {version: blue} → Blue pods get traffic.
   We patch this selector to switch between Blue and Green.
   Port 80 is the external port (what you connect to).
   Port 8000 is the internal port (where Django/gunicorn listens).

django-blue (port 8000 → 8000)
   A DEDICATED service that ALWAYS points to Blue pods only.
   Why? NGINX needs a DNS name to reach Blue pods specifically.
   In Kubernetes, Services create DNS names automatically:
     django-blue.progressive-django.svc.cluster.local
   NGINX uses this DNS name in its upstream config.

django-green (port 8000 → 8000)
   Same as above but for Green pods. NGINX mirrors traffic here.
   DNS: django-green.progressive-django.svc.cluster.local

django-canary-service (port 80 → 8000)
   Points to canary pods (label: app=django-canary).
   Separate from django-service because canary uses different pod labels.
   This lets us test the canary independently.

nginx-shadow-service (port 80 → 80)
   Points to the NGINX proxy pods. When you connect to this service,
   NGINX receives your request, forwards it to Blue, AND silently
   copies it to Green.

WHY ClusterIP TYPE?
   ClusterIP means the service is only reachable INSIDE the cluster.
   That's why we use "kubectl port-forward" to access it from our laptop.
   In production you'd use LoadBalancer or Ingress instead.

SECTION 3: CHECKING PREREQUISITES — COMMAND EXPLANATIONS
================================================================================

COMMAND: minikube status
  WHAT IT DOES: Checks if the local Kubernetes cluster is healthy.
  WHY: Minikube runs Kubernetes inside a Docker container on your machine.
       If apiserver is "Stopped", kubectl commands won't work.
  OUTPUT EXPLAINED:
    host: Running       → The VM/container running Kubernetes is alive
    kubelet: Running    → The agent managing pods on the node is working
    apiserver: Running  → The API server (brain of Kubernetes) is accepting requests
    kubeconfig: Configured → kubectl knows how to connect to this cluster

COMMAND: kubectl get pods -n progressive-django
  WHAT IT DOES: Lists all pods in the "progressive-django" namespace.
  WHY: Pods are the smallest deployable units in K8s. Each pod runs one container.
  FLAGS:
    -n progressive-django → Only show pods in our project's namespace
                            (not system pods from kube-system, etc.)
  OUTPUT COLUMNS:
    NAME    → Pod name (deployment-name + replicaset-hash + pod-hash)
    READY   → "1/1" means 1 container expected, 1 container ready
    STATUS  → "Running" = healthy. "Error"/"CrashLoopBackOff" = broken
    RESTARTS → How many times the container crashed and restarted
    AGE     → How long the pod has been running

COMMAND: kubectl get services -n progressive-django
  WHAT IT DOES: Lists all Services (network endpoints) in our namespace.
  OUTPUT COLUMNS:
    CLUSTER-IP → Internal IP address assigned by Kubernetes (random)
    PORT(S)    → "80/TCP" = listens on port 80, TCP protocol
    TYPE       → ClusterIP = internal only (not exposed outside cluster)

SECTION 4: BLUE-GREEN TEST — COMMAND EXPLANATIONS
================================================================================

COMMAND: kubectl get service django-service -n progressive-django -o jsonpath="{.spec.selector.version}"
  WHAT IT DOES: Reads the "version" field from the service's selector.
  WHY: The selector determines which pods receive traffic.
       If selector says "blue", only pods with label version=blue get traffic.
  FLAGS:
    -o jsonpath="..."  → Extract one specific field from the YAML/JSON output
                         instead of showing the whole service definition.
  OUTPUT:
    "blue" → Traffic goes to Blue (v1) pods
    "green" → Traffic goes to Green (v2) pods

COMMAND: kubectl port-forward service/django-service 8000:80 -n progressive-django
  WHAT IT DOES: Creates a tunnel from your laptop to the Kubernetes service.
  WHY: Services have ClusterIP which is only reachable inside the cluster.
       port-forward maps localhost:8000 on your machine → port 80 on the service.
  FORMAT: port-forward <resource> <local-port>:<remote-port>
    8000 = port on your laptop (http://localhost:8000)
    80   = port on the Kubernetes service
  NOTE: This command KEEPS RUNNING. Use Ctrl+C to stop it.
        You need a SECOND terminal to send test requests.

COMMAND: (Invoke-WebRequest -Uri http://localhost:8000/ -UseBasicParsing).Content
  WHAT IT DOES: PowerShell's equivalent of "curl". Sends HTTP GET request.
  WHY: Tests that the Django app responds correctly through Kubernetes.
  FLAGS:
    -UseBasicParsing → Don't try to parse HTML (faster, works in all PS versions)
    .Content → Extract just the response body (not headers/status)
  OUTPUT EXPLAINED:
    {"message": "Hello from Progressive Delivery Django - v1.0",
     "version": "v1.0",
     "student_demo": "Traffic shifting / rollback / shadow in action"}

    This JSON comes from core/views.py → home() function.
    "version": "v1.0" → VERSION constant is set to "v1.0" in this Docker image.

COMMAND: kubectl patch service django-service -n progressive-django -p "{\"spec\":{\"selector\":{\"version\":\"green\"}}}"
  WHAT IT DOES: Modifies the service's selector LIVE without deleting it.
  WHY: This is the Blue-Green switch! By changing selector from "blue" to "green",
       Kubernetes INSTANTLY routes all new connections to Green pods.
       Old connections to Blue drain naturally. No downtime.
  FLAGS:
    patch → Update a resource in-place (not replace)
    -p '...' → The JSON patch data. We're changing spec.selector.version.
  WHY IT'S ZERO DOWNTIME:
    Both Blue and Green pods are ALREADY running and healthy.
    We're just changing which group the Service points to.
    It's like changing a road sign — the roads (pods) already exist.

OUTPUT AFTER SWITCHING TO GREEN:
  Some requests return: {"version": "v2.0", ...}
  Some requests return: ERROR 500

  WHY ERRORS? In core/views.py, when VERSION == "v2.0":
    if VERSION == "v2.0" and random.random() < 0.15:
        return JsonResponse({"error": "Simulated server error"}, status=500)
  This means ~15% of requests randomly fail. This is INTENTIONAL to simulate
  a buggy v2 release. The canary analysis uses this to detect the problem.

SECTION 5: SHADOW TRAFFIC TEST — COMMAND EXPLANATIONS
================================================================================

COMMAND: kubectl port-forward service/nginx-shadow-service 8080:80 -n progressive-django
  WHAT IT DOES: Tunnels localhost:8080 → NGINX shadow proxy pods.
  WHY 8080: We use a different port than 8000 so it doesn't conflict
            with the Blue-Green port-forward.

WHEN YOU CURL http://localhost:8080/:
  REQUEST FLOW:
    Your laptop → port-forward → NGINX pod → Blue pods (user sees this response)
                                     ↓
                                Green pods (silent copy, response discarded)

  The NGINX config (nginx/nginx.conf) does this:
    location / {
        proxy_pass http://blue;          ← Forward to Blue, return response to user
        mirror /mirror-to-shadow;        ← Also send a COPY to the mirror location
    }
    location = /mirror-to-shadow {
        internal;                        ← Can't be accessed directly from outside
        proxy_pass http://green$request_uri;  ← Send copy to Green
    }

  "mirror" is an NGINX directive that duplicates the request.
  "internal" means this location only works for internal redirects.
  The Green response is DISCARDED — user never sees it.

COMMAND: kubectl logs -n progressive-django -l version=green --tail=15
  WHAT IT DOES: Shows the last 15 lines of log output from Green pods.
  FLAGS:
    -l version=green → Select pods by label (all 3 Green pods)
    --tail=15 → Only show last 15 lines (not the entire log history)
  WHY: To PROVE Green received the mirrored requests.
  OUTPUT EXPLAINED:
    "Internal Server Error: /" → Green's Django processed a mirrored request
    and hit the 15% error simulation. This line in the log proves that Green
    received traffic even though the user only saw Blue's response.
    The user sent 5 requests to NGINX, Blue answered all 5, AND Green also
    received all 5 silently.

SECTION 6: CANARY TEST — COMMAND EXPLANATIONS
================================================================================

COMMAND: kubectl get rollout django-rollout -n progressive-django
  WHAT IT DOES: Shows the Argo Rollout resource (not a regular Deployment).
  WHY "rollout" NOT "deployment": Argo Rollouts uses a custom resource type
       called "Rollout" that adds canary/blue-green strategies on top of
       what regular Kubernetes Deployments can do. Regular Deployments only
       support "RollingUpdate" (replace pods one by one) or "Recreate" (kill
       all, start new). Argo adds percentage-based traffic shifting.
  OUTPUT COLUMNS:
    DESIRED=5   → We want 5 pods total
    CURRENT=5   → 5 pods exist right now
    UP-TO-DATE=5 → 5 pods have the latest template
    AVAILABLE=5  → 5 pods are ready to serve traffic

COMMAND: kubectl patch rollout django-rollout ... --type="json" -p="[{\"op\":\"replace\", ...\"value\":\"django-progressive:v2\"}]"
  WHAT IT DOES: Changes the container image from v1 to v2 in the Rollout spec.
  WHY: This TRIGGERS the canary rollout. Argo detects the image changed and
       starts the canary steps defined in rollout-canary.yaml.
  FLAGS:
    --type="json" → Use JSON Patch format (array of operations)
    "op":"replace" → Replace an existing value
    "path":"/spec/template/spec/containers/0/image" → Path to the container image
       /spec/template/spec/containers/0 = first (only) container in the pod
    "value":"django-progressive:v2" → New image to use

WHAT HAPPENS AFTER TRIGGERING (the canary steps from rollout-canary.yaml):

  Step 1 — setWeight: 20
    Argo scales up 1 new pod with v2 image.
    Now: 4 pods v1 + 1 pod v2 = 20% traffic goes to v2.
    Kubernetes round-robins requests across pods, so 1/5 = 20%.

  Step 2 — pause: 30s
    Argo waits 30 seconds. This is an observation window.
    In production, you'd check dashboards during this pause.

  Step 3 — setWeight: 50
    Argo scales: 3 pods v1 + 2-3 pods v2 ≈ 50% traffic to v2.

  Step 4 — pause: 60s
    Another observation window, longer this time (60s).

  Step 5 — analysis (templateName: error-rate-analysis)
    Argo runs the AnalysisTemplate which queries Prometheus for error rate.
    The query checks: are more than 10% of requests returning 500 errors?
    In our setup, Prometheus is not installed, so the query FAILS TO CONNECT.
    Argo treats connection failures as errors. After 5 consecutive errors
    (consecutiveErrorLimit), Argo ABORTS the rollout.

  Step 6 & 7 — setWeight: 100 + pause: 30s
    These would run IF analysis passed. Since analysis failed, we never reach here.

MONITORING OUTPUT EXPLAINED:

  [01:04:39] Step: 1 | Phase: Paused
    → At step 1 (20% weight). "Paused" = waiting for the 30s timer.

  [01:05:09] Step: 3 | Phase: Paused
    → Jumped to step 3 (50% weight). Steps 1 and 2 completed.
      "Paused" = waiting for the 60s timer.

  [01:06:09] Step: 4 | Phase: Progressing
    → Step 4 (analysis). "Progressing" = analysis is actively running,
      trying to query Prometheus every 30 seconds.

  [01:06:39] Step: 0 | Phase: Degraded
    → ROLLBACK HAPPENED! Step went back to 0. "Degraded" means Argo
      detected a problem and aborted. It automatically:
      1. Scaled DOWN v2 pods (from 2-3 back to 0)
      2. Scaled UP v1 pods (back to 5)
      3. Marked the rollout as "Degraded" (failed)

COMMAND: kubectl describe rollout django-rollout -n progressive-django
  WHAT IT DOES: Shows the full YAML spec + events log for the rollout.
  WHY: The "Events" section at the bottom is a timeline of everything
       that happened during the rollout. Key events:

  ScalingReplicaSet ... from 0 to 1
    → Argo created 1 v2 pod for the 20% canary step.

  ScalingReplicaSet ... from 5 to 4
    → Argo removed 1 v1 pod to make room (total stays at 5).

  RolloutPaused (CanaryPauseStep)
    → Argo paused for the timer (30s or 60s).

  RolloutResumed
    → Timer expired, Argo moves to next step.

  AnalysisRunRunning
    → Argo started the health check analysis.

  AnalysisRunError ... "dial tcp: lookup prometheus ... no such host"
    → The analysis tried to query Prometheus but the DNS name doesn't exist
      in our cluster (we didn't install Prometheus). This is treated as an error.

  RolloutAborted ... Metric "error-rate" assessed Error due to consecutiveErrors (5)
    → After 5 consecutive failures to reach Prometheus, Argo gave up and
      aborted the rollout. This triggers the automatic rollback.

  ScalingReplicaSet ... from 3 to 0  (v2 replicaset)
    → v2 pods are removed.

  ScalingReplicaSet ... from 2 to 5  (v1 replicaset)
    → v1 pods are scaled back to full (5 replicas).

POST-ROLLBACK VERIFICATION:
  All 5 requests return v1.0 with NO errors. This proves:
  1. The rollback was complete — no v2 pods remain in the rotation
  2. The stable version is fully restored automatically
  3. No manual intervention was needed

SECTION 7: KEY KUBERNETES CONCEPTS USED
================================================================================

NAMESPACE (k8s/namespace.yaml):
  A virtual boundary inside Kubernetes. All our resources live in
  "progressive-django" namespace so they don't interfere with other projects.
  -n progressive-django flag tells kubectl to look in that namespace.

DEPLOYMENT (blue-deployment.yaml, green-deployment.yaml):
  Tells Kubernetes: "Run X copies of this container and keep them running."
  If a pod crashes, the Deployment controller automatically creates a new one.
  "replicas: 3" = always maintain exactly 3 running pods.

SERVICE (django-service.yaml):
  A stable network address for a group of pods. Pods come and go (they get
  new IPs when restarted), but the Service IP stays the same. It load-balances
  across all matching pods using round-robin.

ROLLOUT (rollout-canary.yaml):
  Argo's replacement for Deployment. Same basic function (run X pods) but
  adds canary strategy: gradual traffic shifting with pause and analysis steps.

ANALYSISTEMPLATE (analysis-template.yaml):
  Defines HOW to check if the new version is healthy. In our case, it queries
  Prometheus for HTTP error rates. If errors > 10%, the rollout is aborted.

CONFIGMAP (nginx-configmap.yaml):
  Stores configuration data (our nginx.conf) that gets mounted into pods
  as a file. This lets us change NGINX config without rebuilding the image.

PORT-FORWARD:
  Creates a temporary tunnel: your laptop → Kubernetes pod.
  Only for development/testing. In production you'd use LoadBalancer or Ingress.

SECTION 8: THE DJANGO APP — HOW VERSIONS WORK
================================================================================

FILE: django-app/core/views.py

  VERSION = "v1.0"    ← This constant determines the version

  The Docker image django-progressive:v1 was built with VERSION = "v1.0"
  The Docker image django-progressive:v2 was built with VERSION = "v2.0"

  When a request hits /:
    - v1 always returns 200 OK with {"version": "v1.0"}
    - v2 returns 200 OK 85% of the time, but 15% of requests get a 500 error:

      if VERSION == "v2.0" and random.random() < 0.15:
          return JsonResponse({"error": "Simulated server error"}, status=500)

  This 15% error rate is INTENTIONAL. It simulates a buggy release.
  The canary analysis is designed to catch this: if error rate > 10%, rollback.

  When a request hits /healthz:
    - Both v1 and v2 return {"status": "healthy", "version": "v1.0/v2.0"}
    - This endpoint is used by Kubernetes liveness/readiness probes.
    - Kubernetes checks /healthz every few seconds. If it fails, K8s restarts the pod.

SECTION 9: WHY THESE SPECIFIC NUMBERS?
================================================================================

WHY 3 replicas for Blue and Green?
  - Minimum for high availability. If 1 pod dies, 2 still serve traffic.
  - Standard practice for production web apps.
  - 1 pod = single point of failure. 2 pods = still risky. 3 = safe.

WHY 5 replicas for Canary?
  - Allows clean percentage math: 1/5=20%, 2/5=40%, 3/5=60%, 5/5=100%
  - With 3 pods, smallest increment is 33% (too coarse for gradual rollout).
  - In production, you might have 10+ pods for finer control (10%, 20%, etc).

WHY 2 replicas for NGINX?
  - NGINX is lightweight (just proxying, no app logic).
  - 2 gives basic redundancy without wasting resources.
  - 1 would work but has no failover.

WHY port 8000 for Django?
  - Django's default dev server port. Gunicorn uses it too.
  - Convention, not a requirement. Could be any port.

WHY port 80 for Services?
  - Standard HTTP port. Makes the Service URL clean.
  - The Service translates: port 80 (external) → port 8000 (container).
